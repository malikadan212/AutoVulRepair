# Module 2: Fuzz Plan and Harness Lifecycle - Design and Implementation Guide

## Overview

Module 2 automates the transformation from static analysis findings (produced by Module 1) into executable fuzz targets. This module bridges the gap between static vulnerability detection and dynamic fuzzing, creating a fully automated pipeline that generates, builds, executes, and triages fuzzing campaigns.

### Integration with Module 1

Module 2 consumes the output from Module 1:
- **Input**: `static_findings.json` (generated by Module 1's static analysis tools)
- **Output**: Executable fuzz targets, crash artifacts, and triaged vulnerability reports

The workflow follows this pipeline:
```
Module 1 (Static Analysis) 
  → static_findings.json 
  → Module 2 (Fuzz Plan Generator) 
  → fuzz/fuzzplan.json 
  → Module 2 (Harness Generation) 
  → fuzz/targets/*.cc 
  → Module 2 (Build & Execute) 
  → build/* (executables) 
  → Module 2 (Fuzz Execution) 
  → fuzz/artifacts/* (crashes) 
  → Module 2 (Triage) 
  → fuzz/repro/* (reproducible bundles)
```

## Architecture Components

Module 2 consists of six major components that work together to automate the fuzzing lifecycle:

### A. Fuzz-Plan Generator

**Purpose**: Converts static findings into actionable fuzz plans with metadata for harness generation and execution.

**Key Responsibilities**:
- Parse `static_findings.json` from Module 1
- Infer bug classes from rule IDs and messages
- Select appropriate sanitizers based on bug type
- Choose seed directories and dictionaries
- De-duplicate findings targeting the same function
- Generate `fuzz/fuzzplan.json` with target metadata

**Implementation Details**:

**Bug Class Inference Heuristics**:
- **OOB (Out-of-Bounds)**: Detected from rule IDs like `bounds`, `arrayIndexOutOfBounds`, `bufferOverflow`
- **UAF (Use-After-Free)**: Detected from rule IDs like `useAfterFree`, `danglingPointer`, `memoryLeak`
- **Integer-UB (Integer Undefined Behavior)**: Detected from rule IDs like `integerOverflow`, `shiftOverflow`, `signedIntegerOverflow`
- **Null-Deref**: Detected from rule IDs like `nullPointer`, `nullDereference`
- **Format-String**: Detected from rule IDs like `formatString`, `printfFormat`

**Sanitizer Selection Logic**:
- **ASan (AddressSanitizer)**: Selected for memory-related bugs (OOB, UAF, Null-Deref)
- **UBSan (UndefinedBehaviorSanitizer)**: Selected for integer/shift/arithmetic UB bugs
- **Combined**: Both sanitizers used when bug class is ambiguous or multiple bug types detected

**Seed and Dictionary Selection**:
- **Parser-type bugs**: Include `fuzz/auto.dict` with common parser tokens
- **API-type bugs**: Include structured input seeds from `fuzz/seeds/api/`
- **Protocol bugs**: Include protocol-specific dictionaries from `fuzz/seeds/protocol/`
- **Default**: Generic byte-stream seeds from `fuzz/seeds/generic/`

**De-duplication Strategy**:
- Findings are grouped by `<file-stem>::<function>` (e.g., `parse.cc::ParseFrame`)
- Only the highest-severity finding per target is retained
- Confidence levels are aggregated (highest confidence wins)

**Fuzz Plan JSON Structure**:
```json
{
  "version": "1.0",
  "generated_at": "2024-01-15T10:30:00Z",
  "targets": [
    {
      "target_id": "parse_cc_ParseFrame",
      "source_file": "demo/parse.cc",
      "file_stem": "parse",
      "function_name": "ParseFrame",
      "bug_class": "OOB",
      "rule_id": "bounds",
      "severity": "high",
      "confidence": "high",
      "line_number": 42,
      "sanitizers": ["address", "undefined"],
      "seed_directories": ["fuzz/seeds/parser/"],
      "dictionaries": ["fuzz/auto.dict"],
      "priority": 8.5,
      "harness_type": "template",
      "harness_template": "bytes_to_api"
    }
  ],
  "metadata": {
    "total_findings": 15,
    "deduplicated_targets": 8,
    "bug_class_breakdown": {
      "OOB": 5,
      "UAF": 2,
      "Integer-UB": 1
    }
  }
}
```

### B. Toolbox-Based Harness Generation

**Purpose**: Generate fuzzing harnesses using a toolbox approach that selects the appropriate generation method based on target characteristics.

**Toolbox Components**:

#### 1. Template-Based Harnesses
**Use Case**: Common bytes-to-API pattern for parsers and raw-inlet functions

**Template Structure**:
```cpp
#include <fuzzer/FuzzedDataProvider.h>
#include "target_header.h"

extern "C" int LLVMFuzzerTestOneInput(const uint8_t *data, size_t size) {
    FuzzedDataProvider fdp(data, size);
    
    // Template-specific harness code
    // Generated based on function signature and bug class
    
    return 0;
}
```

**Template Types**:
- `bytes_to_api`: Direct byte stream to function call
- `fdp_adapter`: FuzzedDataProvider for typed parameters
- `parser_wrapper`: Parser-specific input handling
- `api_sequence`: Stateful API sequence harnesses

#### 2. FuzzGen / Automated Adapters
**Use Case**: API-like functions that take structured parameters

**Approach**: 
- Inspect function signatures from source code
- Generate harness code that synthesizes typed parameters
- Handle complex data structures automatically

#### 3. FuzzedDataProvider Adapters
**Use Case**: Functions expecting integers, flags, or strings

**Implementation**:
```cpp
FuzzedDataProvider fdp(data, size);
int param1 = fdp.ConsumeIntegral<int>();
bool flag = fdp.ConsumeBool();
std::string str = fdp.ConsumeRandomLengthString();
target_function(param1, flag, str);
```

#### 4. Domain-Specific Generators
**Use Case**: Stateful components, protocol parsers, structured input drivers

**Examples**:
- Protocol frame generators (preserve header structure)
- State machine harnesses (sequence of API calls)
- Grammar-aware generators (preserve syntax rules)

**Harness Selection Logic**:
```python
def select_harness_type(target_metadata):
    if target_metadata['bug_class'] == 'OOB' and 'parser' in target_metadata['function_name'].lower():
        return 'parser_wrapper'
    elif target_metadata['harness_type'] == 'api':
        return 'fuzzgen_adapter'
    elif target_metadata['function_signature']['param_count'] <= 3:
        return 'fdp_adapter'
    else:
        return 'bytes_to_api'
```

**Harness Naming Convention**:
- Format: `fuzz_<file_stem>_<function_name>.cc`
- Example: `fuzz_parse_ParseFrame.cc`
- All harnesses stored under `fuzz/targets/`

**Metadata Logging**:
- Each harness generation logs: template used, generation timestamp, target metadata
- Stored in `fuzz/targets/.metadata.json` for traceability

### C. Build Orchestration

**Purpose**: Compile each harness + target source pair into standalone fuzzing executables with proper sanitizers and optimization.

**Build Configuration**:

**Compiler Flags**:
```bash
clang++ \
  -fsanitize=fuzzer,address,undefined \
  -g \
  -O1 \
  -std=c++17 \
  -I./include \
  -o build/fuzz_parse_ParseFrame \
  fuzz/targets/fuzz_parse_ParseFrame.cc \
  demo/parse.cc \
  -lFuzzingEngine
```

**Flag Breakdown**:
- `-fsanitize=fuzzer`: Links libFuzzer runtime (provides `main()` and fuzzing engine)
- `-fsanitize=address`: Enables AddressSanitizer for memory error detection
- `-fsanitize=undefined`: Enables UndefinedBehaviorSanitizer for integer/shift UB
- `-g`: Debug symbols for readable stack traces
- `-O1`: Modest optimization (balance between performance and trace quality)

**Build Process**:
1. Iterate through all targets in `fuzz/fuzzplan.json`
2. For each target:
   - Locate harness file: `fuzz/targets/fuzz_<file_stem>_<function>.cc`
   - Locate source file: from `target.source_file` in fuzz plan
   - Resolve dependencies (headers, libraries)
   - Compile with appropriate sanitizers
   - Place executable in `build/fuzz_<file_stem>_<function>`
3. Log build results (success/failure) to `build/.build_log.json`

**Dependency Resolution**:
- Parse `#include` directives in harness and source files
- Resolve include paths from project structure
- Handle library dependencies (if any)

**Build Script Structure**:
```python
def build_fuzz_target(target_metadata, project_root):
    harness_path = f"fuzz/targets/fuzz_{target_metadata['file_stem']}_{target_metadata['function_name']}.cc"
    source_path = target_metadata['source_file']
    output_name = f"build/fuzz_{target_metadata['file_stem']}_{target_metadata['function_name']}"
    
    sanitizers = ','.join(target_metadata['sanitizers'])
    
    cmd = [
        'clang++',
        f'-fsanitize=fuzzer,{sanitizers}',
        '-g', '-O1', '-std=c++17',
        f'-I{project_root}/include',
        f'-o{output_name}',
        harness_path,
        source_path
    ]
    
    result = subprocess.run(cmd, capture_output=True, text=True)
    return result.returncode == 0
```

### D. Fuzz Execution

**Purpose**: Execute fuzzing campaigns for each target with proper configuration, corpus management, and artifact collection.

**Execution Configuration**:

**Per-Target Execution**:
```bash
./build/fuzz_parse_ParseFrame \
  -artifact_prefix=fuzz/artifacts/parse_ParseFrame_ \
  -runs=1000000 \
  -max_total_time=3600 \
  fuzz/corpus/parse_ParseFrame/ \
  fuzz/seeds/parser/
```

**Runtime Sanitizer Options**:
```bash
export ASAN_OPTIONS="detect_stack_use_after_return=true:check_initialization_order=true:strict_init_order=true:halt_on_error=true:abort_on_error=true"
export UBSAN_OPTIONS="print_stacktrace=1:halt_on_error=1:abort_on_error=1"
export MSAN_OPTIONS="print_stats=1:halt_on_error=1"
```

**Sanitizer Option Breakdown**:
- `halt_on_error=true`: Immediate halting on detected errors (no continuation)
- `abort_on_error=true`: Abort process immediately (clean crash state)
- `print_stacktrace=1`: Generate readable stack traces
- `detect_stack_use_after_return=true`: Enhanced stack detection

**Directory Structure**:
```
fuzz/
├── artifacts/          # Crash artifacts (minimized inputs)
│   ├── parse_ParseFrame_crash-<hash>
│   └── parse_ParseFrame_timeout-<hash>
├── corpus/            # Fuzzing corpus (coverage-increasing inputs)
│   ├── parse_ParseFrame/
│   └── api_ProcessRequest/
└── seeds/             # Initial seed inputs
    ├── parser/
    ├── api/
    └── generic/
```

**Execution Orchestration**:
1. For each target in fuzz plan:
   - Create corpus directory if it doesn't exist
   - Set sanitizer environment variables
   - Execute fuzzer with timeout (default: 1 hour per target)
   - Monitor for crashes/timeouts
   - Collect artifacts to `fuzz/artifacts/`
   - Update corpus in `fuzz/corpus/<target>/`
2. Log execution results to `fuzz/.execution_log.json`

**Parallel Execution**:
- Multiple targets can be fuzzed in parallel (limited by CPU cores)
- Each fuzzer runs in isolated process with resource limits
- Resource limits: CPU time, memory (2GB per fuzzer)

### E. Custom Mutation

**Purpose**: Enhance libFuzzer's mutation strategy with domain-aware transformations that improve discovery rates for protocol and parser bugs.

**Implementation**:

**Custom Mutator Interface**:
```cpp
#include <fuzzer/FuzzedDataProvider.h>

extern "C" size_t LLVMFuzzerCustomMutator(
    uint8_t *Data, size_t Size, size_t MaxSize, unsigned int Seed) {
    
    // Custom mutation logic
    // Preserve protocol structures while mutating data
    
    return NewSize;
}
```

**Mutation Strategies**:

1. **Length-Prefix Preservation**:
   - Detect two-byte length fields (big-endian/little-endian)
   - When mutating payload, update length field to match
   - Prevents invalid length crashes that mask real bugs

2. **Control-Byte Targeting**:
   - Identify control bytes (opcodes, flags, type fields)
   - Bias mutations toward control regions
   - Flip bits in control bytes more frequently

3. **Grammar-Preserving Insertions**:
   - For structured formats, insert valid field boundaries
   - Preserve field delimiters during mutations
   - Maintain basic structure while exploring new inputs

4. **Protocol-Aware Transformations**:
   - Recognize common protocol patterns (headers, checksums)
   - Mutate payload while preserving header structure
   - Update checksums if protocol requires

**Integration with libFuzzer**:
- Mutator compiled as separate `.so` (shared object) or linked statically
- Loaded via `-custom_mutator=<path>` flag or environment variable
- Falls back to default mutations if custom mutator unavailable

**Usage**:
```bash
# With custom mutator
./build/fuzz_parse_ParseFrame -custom_mutator=./mutators/protocol_mutator.so ...

# Without (default libFuzzer mutations)
./build/fuzz_parse_ParseFrame ...
```

**Comparison Mode**:
- System can run targets with and without custom mutator
- Compare crash discovery rates
- Log mutation effectiveness metrics

### F. Repro-Kit Production and Triage

**Purpose**: Automatically minimize crash inputs, create reproducible bundles, and extract crash metadata for triage and fix classification.

**Crash Minimization**:

**Process**:
```bash
# Minimize crash input
./build/fuzz_parse_ParseFrame -minimize_crash=1 \
  fuzz/artifacts/parse_ParseFrame_crash-abc123
```

**Output**: Minimized input saved to `fuzz/artifacts/parse_ParseFrame_crash-abc123.minimized`

**Repro-Kit Structure**:
```
fuzz/repro/
└── parse_ParseFrame_crash-abc123/
    ├── poc.bin              # Minimized proof-of-crash input
    ├── run.sh               # One-liner reproduction script
    ├── log.txt              # Crash log with stack trace
    ├── metadata.json        # Extracted crash metadata
    └── README.md            # Human-readable description
```

**Reproduction Script (`run.sh`)**:
```bash
#!/bin/bash
# Reproduce crash: parse_ParseFrame_crash-abc123
export ASAN_OPTIONS="halt_on_error=1:abort_on_error=1"
export UBSAN_OPTIONS="print_stacktrace=1:halt_on_error=1"
./build/fuzz_parse_ParseFrame poc.bin
```

**Triage and Metadata Extraction**:

**Crash Type Classification**:
- **ASan Errors**:
  - `heap-buffer-overflow`: Read/write beyond heap allocation
  - `stack-buffer-overflow`: Read/write beyond stack allocation
  - `use-after-free`: Access to freed memory
  - `double-free`: Double free of same pointer
  - `memory-leak`: Memory not freed (informational)

- **UBSan Errors**:
  - `integer-overflow`: Signed integer overflow
  - `unsigned-integer-overflow`: Unsigned overflow (informational)
  - `shift-overflow`: Shift amount >= bit width
  - `null-pointer-dereference`: Null pointer access

**Metadata Extraction**:
```json
{
  "crash_id": "parse_ParseFrame_crash-abc123",
  "target": "parse_cc_ParseFrame",
  "crash_type": "heap-buffer-overflow",
  "sanitizer": "address",
  "crashing_location": {
    "file": "demo/parse.cc",
    "function": "ParseFrame",
    "line": 42,
    "column": 15
  },
  "stack_trace": [
    {"file": "demo/parse.cc", "function": "ParseFrame", "line": 42},
    {"file": "demo/parse.cc", "function": "ProcessInput", "line": 28},
    {"file": "fuzz/targets/fuzz_parse_ParseFrame.cc", "function": "LLVMFuzzerTestOneInput", "line": 15}
  ],
  "input_size": 128,
  "minimized_size": 12,
  "dictionary_strings": ["FRAME", "LEN"],
  "suggested_fix_classification": {
    "patch_type": "bounds_check",
    "confidence": "high",
    "description": "Add bounds check before buffer copy at line 42"
  },
  "reproduced_at": "2024-01-15T11:30:00Z"
}
```

**Fix Classification**:

**Patch Type Inference**:
- **bounds_check**: OOB errors → Add bounds validation
- **length_validation**: Integer overflow → Validate length fields
- **lifetime_management**: UAF errors → Fix pointer lifetime
- **null_check**: Null dereference → Add null pointer checks
- **type_validation**: Type confusion → Add type checks

**Classification Logic**:
```python
def classify_fix(crash_metadata):
    crash_type = crash_metadata['crash_type']
    bug_class = crash_metadata['bug_class']
    
    if crash_type in ['heap-buffer-overflow', 'stack-buffer-overflow']:
        return 'bounds_check'
    elif crash_type == 'use-after-free':
        return 'lifetime_management'
    elif crash_type == 'integer-overflow':
        return 'length_validation'
    elif crash_type == 'null-pointer-dereference':
        return 'null_check'
    else:
        return 'unknown'
```

   - Collect all repro-kits
   - Generate summary report
   - Update database with fuzzing results

- **Resource Limits**: CPU time, memory limits per fuzzer
- **Input Validation**: Validate all inputs from Module 1
- **Path Traversal Prevention**: Sanitize file paths in harness generation
- **Code Injection Prevention**: Template engine should not execute arbitrary code
