- **Module 4 (Build Orchestrator)** ✅
  - **What:** Reads compiled fuzz binaries
  - **Where:** `self.build_dir = os.path.join(scan_dir, 'build')`

### Indirect Dependencies:
- **Module 3 (Harness Generator)** ⚠️
  - **What:** Target names come from harness filenames
  - **Pattern:** Looks for files starting with `fuzz_*`
  - **File:** `src/fuzz_exec/executor.py` line 48
  - **Why:** Naming convention links harnesses to executables

## Cross-Module Data Dependencies

### Shared Data Structures

#### 1. **scan_id** (Universal Key)
- **Used by:** ALL modules
- **Purpose:** Links all artifacts together
- **Format:** UUID string
- **Example:** `"5d3afbbb-f4a6-4702-81c4-b51201f6bc5c"`

#### 2. **target_name** (Naming Convention)
- **Created by:** Module 3 (Harness Generator)
- **Format:** `fuzz_{file_stem}_{function_name}`
- **Used by:**
  - Module 4: Names compiled binaries
  - Module 5: Identifies which target to run
  - Module 6: Associates crashes with targets
  - Module 7: References in reproduction kits

#### 3. **Bug Class** (Classification)
- **Inferred by:** Module 2 (Fuzz Plan)
- **Used by:**
  - Module 3: Selects harness template
  - Module 4: Chooses sanitizers
  - Module 6: Validates crash type matches prediction

#### 4. **Sanitizer Output** (Error Messages)
- **Generated by:** Module 4 (Build - sanitizer instrumentation)
- **Captured by:** Module 5 (Fuzz Execution)
- **Analyzed by:** Module 6 (Crash Triage)
- **Referenced by:** Module 7 (Repro Kits)

## Feedback Loops

### 1. **Validation Loop: Module 6 → Module 2**
```
Module 2 (Fuzz Plan) predicts bug class
    ↓
Module 5 (Fuzz Execution) finds crashes
    ↓
Module 6 (Crash Triage) confirms/refutes prediction
    ↓
[Feedback] Validates static analysis accuracy
```

**Purpose:** Measure how accurate static analysis predictions are

**Example:**
- Fuzz plan says: "Buffer-Overflow" (from static analysis)
- Crash triage finds: "Heap Buffer Overflow"
- ✅ Prediction confirmed!

---

### 2. **Refinement Loop: Module 7 → Module 1**
```
Module 1 (Static Analysis) finds vulnerability
    ↓
Module 7 (Repro Kit) generates patch suggestion
    ↓
[Feedback] Patch references original static finding
```

**Purpose:** Link dynamic findings back to static findings

**Example:**
- Static finding: Line 42, buffer.c, "Array index out of bounds"
- Repro kit patch: "Add bounds check at line 42 in buffer.c"

## File System Dependencies

### Shared Directory Structure
```
scans/{scan_id}/
├── source/              ← Module 1 creates, Module 2 reads
├── artifacts/           ← Module 1 writes
├── static_findings.json ← Module 1 writes, Module 2 reads
├── fuzz/
│   ├── fuzzplan.json   ← Module 2 writes, Module 3 reads
│   ├── harnesses/      ← Module 3 writes, Module 4 reads
│   ├── results/        ← Module 5 writes, Module 6 reads
│   ├── crashes/        ← Module 5 writes, Module 6 & 7 read
│   └── triage/         ← Module 6 writes, Module 7 reads
├── build/              ← Module 4 writes, Module 5 reads
└── repro_kits/         ← Module 7 writes
```

| `source/` | Module 1 | Module 2 | Signature extraction |
| `static_findings.json` | Module 1 | Module 2 | Fuzz plan input |
| `fuzzplan.json` | Module 2 | Module 3 | Harness generation |

| `build/fuzz_*` | Module 4 | Module 5 | Executables to run |
| `crashes/crash-*` | Module 5 | Module 6, 7 | Crash artifacts |

| `triage_results.json` | Module 6 | Module 7 | Severity/exploitability |

## Configuration Dependencies

### Sanitizer Selection Chain
```
Module 2 (Fuzz Plan)
    ↓ (selects sanitizers based on bug class)
Module 4 (Build Orchestrator)
    ↓ (applies sanitizers during compilation)
Module 5 (Fuzz Execution)
    ↓ (sanitizers detect crashes at runtime)
Module 6 (Crash Triage)
    ↓ (parses sanitizer output)
```

**Example:**
- Module 2: Bug class = "Buffer-Overflow" → Sanitizers = ["ASan", "UBSan"]
- Module 4: Compiles with `-fsanitize=address,undefined`
- Module 5: ASan detects heap-buffer-overflow
- Module 6: Parses "heap-buffer-overflow" from ASan output

```

    ↓

    ↓

```
